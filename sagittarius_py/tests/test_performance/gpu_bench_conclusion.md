## 3. Key Findings

### 3.1 Crossover Point
The GPU-accelerated solver begins to outperform the CPU-based solver at approximately **\$N=14\$** (Basis Dimension \$\approx 1,000\$). Below this threshold, the overhead of CUDA kernel launches and memory management dominates.

### 3.2 Impact of Operator Caching
A critical optimization was implemented in \`Sagittarius.jl/src/physics.jl\` to reuse \`ReducedRydbergOperator\` objects and cache the \`CuSparseMatrixCSC\` representation. Without this caching, Host-to-Device transfer and matrix conversion overhead prevented speedups even at \$N=20\$.

### 3.3 Scaling Behavior
The GPU execution time scales very favorably with basis size. At \$N=20\$, the GPU remains under-utilized, suggesting that the RTX 5070 Ti can handle significantly larger simulations (estimated \$N \approx 25-28\$) before reaching compute or memory saturation.

## 4. Recommendations for Future Optimization
1.  **Matrix-Free GPU Kernels**: Implementing bitwise Hamiltonian application directly in CUDA kernels would eliminate the memory overhead of sparse matrices, allowing simulations of up to \$N \approx 40\$ on 16GB VRAM.
2.  **GPU-Native Observables**: Pre-calculating and caching bitmasks on the device will further reduce scalar sync points during the integration loop.
3.  **Adaptive Backend Selection**: The \`Simulation\` class should automatically default to CPU for \$N < 14\$ and GPU for \$N \ge 14\$ to optimize for overall user experience.
